SSE Event Loss on Railway

Bug Repro

The Problem

When a NestJS app deployed on Railway sends SSE (Server-Sent Events) responses, events are silently dropped by Railway's proxy layer when the stream is closed immediately after sending a large event (~250KB+). The proxy drops any buffered data that hasn't been flushed to the client when it receives the upstream TCP FIN.

What gets dropped

The server sends three SSE events then closes the stream:

event: init        ~50 bytes    ✔ received
event: preview     ~250 KB      ✔ received
event: complete    ~50 bytes    ✘ LOST     (stream closed immediately after)

The complete event never reaches the client. The connection errors out instead. This does not happen when running locally — only through Railway's proxy.

Root cause: close race, not buffering

Testing confirmed this is not a general SSE buffering issue. When the stream stays open (no close), small events arrive fine after large ones. The issue only occurs when response.end() (TCP FIN) is sent immediately after a large write — the proxy closes the downstream connection before flushing all buffered data.

Key observations

  • Adding a 500ms delay before closing the stream makes all events arrive
  • Without close: 250KB → 1KB messages both arrive (PASS)
  • With close: 250KB arrives, 1KB is lost (FAIL)
  • Small payloads (~10KB) don't trigger the issue — only large ones (~250KB+)
  • The issue is 100% reproducible, not intermittent
  • The NestJS @Sse() decorator correctly sets X-Accel-Buffering: no, Cache-Control: no-transform, and Connection: keep-alive

Environment

  • NestJS 10.3.9 with @Sse() decorator
  • RxJS Subject piped as Observable
  • Railway deployment with Railpack builder
  • Response headers include: x-railway-cdn-edge, x-served-by

Tests

Two tests that isolate the issue to the connection close behavior:

Test 1 — Close Race

Reproduces the bug

Sends init + large preview + complete, then immediately closes the stream. The complete event is consistently lost through Railway's proxy. Adding a delay before close fixes it.
Test 2 — Message Sequence

Proves it's not a buffering issue

Sends configurable sequences of large/small messages. Without close: all messages arrive. With close: messages after the large one are lost. Confirms the issue is the FIN racing with buffered data.
