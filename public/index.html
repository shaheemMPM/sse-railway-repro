SSE Event Loss on Railway

Bug Repro

The Problem

When a NestJS app deployed on Railway sends SSE (Server-Sent Events) responses, events can be silently dropped by Railway's proxy layer (Fastly CDN). This happens when a small SSE event is sent immediately after a large one (~250KB+), and the stream is closed shortly after.

What gets dropped

The server sends three SSE events in sequence:

event: init        ~50 bytes    ✔ received
event: preview     ~250 KB      ✔ received
event: complete    ~50 bytes    ✘ LOST

The complete event never reaches the client. The connection errors out instead. This does not happen when running locally — only through Railway's proxy.

Key observations

  • Adding a 500ms delay before closing the stream makes all events arrive
  • Small payloads (~10KB) don't trigger the issue — only large ones (~250KB+)
  • The issue is 100% reproducible, not intermittent
  • The NestJS @Sse() decorator correctly sets X-Accel-Buffering: no, Cache-Control: no-transform, and Connection: keep-alive

Environment

  • NestJS 10.3.9 with @Sse() decorator
  • RxJS Subject piped as Observable
  • Railway deployment with Railpack builder
  • Response headers show: x-railway-cdn-edge: fastly/...

Tests

Two tests to isolate whether the issue is a close race (FIN vs data) or a general buffering problem:

Scenario A

Close Race Test

Sends init + large preview + complete, then immediately closes the stream. Tests if the TCP FIN races with the last event through the proxy.
Scenario B

Message Sequence Test

Sends configurable sequences of large/small messages without closing the stream. Tests if small events get lost after large ones mid-stream.
